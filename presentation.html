Project Overview

What is Campus Cafe?

Campus Cafe is a point-of-sale (POS) system designed for a campus dining environment. It supports two user roles — customers who browse menus and place orders, and staff who manage inventory and review session summaries.

Customer Features

  • Browse menu by category
  • Add / remove items from cart
  • Daily special with automatic discount
  • Checkout with loyalty points
  • Printed receipt with tax breakdown

Staff Features

  • Secure login (ID + password)
  • View real-time inventory levels
  • Restock items
  • Session summary on quit (orders + revenue)

Dual Interface

  • Terminal-based CLI (main.py)
  • React web application (frontend)
  • Both interfaces share the same backend logic

Data Persistence

  • Excel files for all data storage
  • Orders, customers, inventory, staff
  • Accumulated loyalty points tracked

Key decision: We check stock per unit in a loop rather than checking total availability upfront. This ensures partial fulfillment — if 3 are requested but only 2 remain, the customer gets 2.

checkout() — receipt & loyalty points
def checkout():
    subtotal = order.get_total()
    total_before_tax = order.get_total_after_discount()
    discount = subtotal - total_before_tax
    tax = order.TAX_RATE * total_before_tax
    total = total_before_tax + tax

    # Loyalty points
    customers = Customers()
    if customers.check_customer(telephone):
        customer = customers.get_customer(telephone)
        customer.earn_points(total)
        customers.update_customer(customer)
    else:
        customer = Customer(telephone)
        customer.earn_points(total)
        customers.add_customer(customer)

    # Print receipt, save order, track session
    order.save_to_excel()
    session_orders[len(session_orders)] = {
        "Total": total, "Items": str(order)
    }

Class Design Deep Dive

Object-oriented design for Menu, Order, Inventory, Customer, and Staff

Class: Menu & MenuItem

backend/menu.py — Menu management with daily specials

Design Decisions

  • MenuItem is a simple data class — name, price, category. Keeps things lightweight.
  • Menu loads all items from Menu.xlsx on init, so data is always fresh from disk.
  • Daily Special is randomly chosen via random.choice() each session, with a configurable discount rate (default 20%).
  • Fuzzy search: get_item() ignores case and spaces, so "fruitsalad" matches "Fruit Salad".
MenuItem & Menu (key methods)
class MenuItem:
    def __init__(self, name, price, category):
        self.name = name
        self.price = price
        self.category = category

class Menu:
    def __init__(self, filename="Menu.xlsx",
                 rate=0.2):
        self.menu = self.load_from_excel()
        self.special_item = self.daily_special()
        self.special_discounted_rate = rate

    def daily_special(self):
        return random.choice(self.menu)

    def get_item(self, name):
        for item in self.menu:
            if item.name.lower().replace(" ","") \
               == name.lower().replace(" ",""):
                return item
        return None

Class: Order

backend/order.py — Cart management, pricing, and tax calculation

Design Decisions

  • Items as a list: We store each item instance individually (not a dict), making add/remove simple with append() / remove().
  • Discount logic lives in get_total_after_discount() — only the daily special item gets the discount applied per unit.
  • Tax rate is a class constant (0.1 = 10%), easy to adjust.
  • save_to_excel() auto-increments order_id and appends to the Orders spreadsheet with timestamps.
Order (key methods)
class Order:
    def __init__(self, menu):
        self.items = []
        self.menu = menu
        self.TAX_RATE = 0.1

    def get_total_after_discount(self):
        total = 0.0
        for item in self.items:
            if item == self.menu.special_item:
                total += item.price * \
                  (1 - self.menu.special_discounted_rate)
            else:
                total += item.price
        return total

    def get_items_num(self):
        # Returns {MenuItem: count, ...}
        item_counter = {}
        for item in self.items:
            if item not in item_counter:
                item_counter[item] = 1
            else:
                item_counter[item] += 1
        return item_counter

Class: Inventory

backend/inventory_ShuranRyu.py — Contributed by teammate Shuran Ryu

Collaborative Development

The Inventory class was written by our classmate Shuran Ryu and merged into our codebase. This demonstrates our ability to integrate external code into a cohesive system, aligning interfaces so that main.py and other classes interact with it seamlessly.

Key Methods

  • check_stock(item) → returns True/False — safe guard before deducting.
  • deduct_stock(item) — decrements by 1 unit per call.
  • restore_stock(item) — increments by 1 when an order item is deleted.
  • restock(name, amount) — staff can bulk-add any quantity.
Inventory (key methods)
class Inventory:
    def __init__(self, filename="Inventory.xlsx"):
        self.stock = self.load_from_excel()
        # stock = {"Americano": 5, ...}

    def check_stock(self, item):
        return self.stock.get(item.name, 0) > 0

    def deduct_stock(self, item):
        if self.check_stock(item):
            self.stock[item.name] -= 1

    def restore_stock(self, item):
        self.stock[item.name] += 1

    def restock(self, item_name, amount):
        self.stock[item_name] = \
            self.stock.get(item_name, 0) + amount

Class: Customer & Customers

backend/customer.py — Loyalty program with automatic point tracking

Design Decisions

  • Auto-generated IDs: A class-level counter produces unique 9-digit IDs (e.g., 000000001). On load, the counter syncs with the max existing ID.
  • Phone as identifier: Customers are looked up by phone number; duplicate prevention is built in.
  • Points policy: Earn 1 point per $5 spent. Clear, simple, and configurable.
  • Customers class manages the collection — CRUD operations + Excel persistence.
Customer (key methods)
class Customer:
    counter = 0  # Class-level auto-increment

    def __init__(self, telephone, name="",
                 points=0, id=None):
        if id is not None:
            self.customer_id = id
        else:
            Customer.counter += 1
            self.customer_id = \
                str(Customer.counter).zfill(9)

        self.telephone = telephone
        self.points = points
        self.earn_point_policy = 5

    def earn_points(self, consuming):
        # 1 point per $5 spent
        self.points += consuming // 5

Class: Staff & Staffs

backend/staff.py — Secure staff authentication and management

Design Decisions

  • Same ID pattern as Customer — 9-digit auto-increment with class-level counter.
  • Password-protected: Staff must enter both ID and password to access the system. In the CLI, getpass hides password input.
  • Staffs collection mirrors the Customers pattern: check, get, add, delete, update + Excel I/O.

Consistent patterns: Customer and Staff follow the same architectural template — entity class + collection class + Excel persistence. This makes the codebase predictable and easy to extend.

Special Feature: Loyalty Points Tracking

Every checkout automatically identifies returning customers by phone number, accumulates their loyalty points in Customers.xlsx, and displays the updated balance on the receipt. New customers are auto-registered on first purchase with a unique 9-digit ID. All data persists across sessions.

React Frontend

Modern web interface for the Campus Cafe POS

To make our project more accessible and visually engaging, we built a full web application using React. A FastAPI server (app.py) wraps the exact same backend classes, exposing them as REST API endpoints.

Tech Stack

  • React 18 + Vite
  • React Router v6
  • FastAPI (Python)
  • Vite proxy for API calls

Customer Pages

  • Home — landing page
  • Menu — browse by category
  • Cart — review & edit
  • Checkout — pay & receipt

Staff Pages

  • Staff Login — secure auth
  • Inventory Management
  • Restock items
  • Session summary on quit

Same Backend, Two Interfaces

Both the terminal CLI and the React frontend call the same Python classes. The web app adds a visual layer on top without duplicating any business logic. This demonstrates clean separation of concerns — the backend handles data and rules, while the frontend handles presentation.
